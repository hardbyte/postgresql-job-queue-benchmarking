# oban-bench adapter audit

Sub-agent (Explore) research report from 2026-05-09T13:45Z.
Oban version constraint `~> 2.18` resolved to `2.22.1` (oban-bench/mix.lock).
Tooling: file:line citations against oban-bench at HEAD.

## Findings

### 1. Producer bulk-insert path — documented API used
`oban-bench/lib/oban_bench/long_horizon.ex:249-283` uses `Oban.insert_all/2`
(documented at https://hexdocs.pm/oban/Oban.html#insert_all/2) when
`producer_batch_max > 1`, issuing a single multi-row INSERT per batch. Single-row
path at line 249–261 uses `Oban.insert/1` when `batch_max <= 1`. Default
`PRODUCER_BATCH_MAX=1` (line 46) means the benchmark historically runs row-at-a-time,
but can be overridden at runtime. Both paths are documented and correct.

### 2. Consumer path — single supervisor, single worker pool
`oban-bench/config/config.exs:25` statically declares three queues:
`benchmark: 50, chaos: 10, long_horizon_bench: 32`. Oban's documented architecture
(README.md, lines 82–85) runs each queue in isolation with its own concurrency limit.
`oban-bench/lib/oban_bench.ex:39,119,143` call `Oban.scale_queue(queue: :benchmark,
limit: worker_count)` to dynamically adjust. Application supervisor (application.ex:6–11)
starts a single Oban instance, which internally manages all queues via one supervisor tree.
This is the documented surface; no per-replica consumer isolation (e.g., multiple Oban
instances per replica) is configured.

### 3. Retry / DLQ handling — completes only, discarded never exercised
Workers (`oban_bench/lib/oban_bench/worker.ex:5` and `chaos_worker.ex:5`) define
`max_attempts: 3` (line 2) and `max_attempts: 5` (chaos, line 7). Jobs always return
`:ok` (lines 5–7 and 13–18), meaning success, never `:error` or `:discard`. Completion
counter (`long_horizon.ex:500`) only increments on `:ok`. No DLQ jobs (state `discarded`)
are generated by this benchmark. Lifeline plugin (`config/config.exs:27`) will rescue
orphaned jobs after `rescue_after_secs` (default 15 s), but no in-benchmark failure path
exercises it. The `discard` return value from workers is documented (Worker.ex, lines
379–380) but unexercised.

### 4. Chaos resilience — relies on Oban supervisor recovery
No explicit reconnection logic in oban-bench. If the database connection is lost,
Oban's internal connection pooling (via Postgrex and Ecto's db_connection,
mix.lock lines 2, 8) handles reconnection through normal retry backoff. The producer
loop (`long_horizon.ex:250–260`) catches `:exit` and `ArgumentError` but does not
implement explicit socket-loss detection. Instead, the benchmark relies on Oban's
supervisor tree and Postgrex's built-in reconnection; a DB restart will eventually
re-establish the pool connection without benchmark-level intervention.

### 5. Per-replica isolation — single shared queue, observer-only per-instance split
All replicas subscribe to the same queue names (`benchmark`, `chaos`, `long_horizon_bench`),
configured in `config/config.exs:25`. Jobs are claimed and executed by *any* available
replica's workers. Multi-replica scaling is horizontal concurrency, not isolated queues.
Observer metrics (`long_horizon.ex:31, 426–431`) are only emitted by instance 0
(via `observer_enabled?()`, line 431) to avoid duplicate telemetry; non-zero replicas
idle the depth-polling loop (`long_horizon.ex:322–325`) but still execute jobs. This is
correct multi-replica semantics, not a bottleneck.

### 6. Documented knobs unset, using defaults
`config/config.exs:5–9` exposes `MAX_CONNECTIONS` (default 20) and `RESCUE_AFTER_SECS`
(default 15). Oban also supports `dispatch_cooldown` (default 5 ms), `stage_interval`
(default 5 s), `shutdown_grace_period` (default 30 s) — none overridden in config.
`long_horizon.ex:46–47` exposes `PRODUCER_BATCH_MAX` (default 1) and `PRODUCER_BATCH_MS`
(default 10); these control bulk-insert batching. Running the benchmark without explicit
overrides uses sensible defaults matching Oban's production mode.

### 7. Missing Pro features — only Basic engine, no Smart, no Workflows/Batches
`config/config.exs:23` explicitly declares `engine: Oban.Engines.Basic`, the default
non-Pro option. README.md (lines 136–148) documents Oban Pro's Smart Engine (global
concurrency, auto batching), Workflows (job composition), Batches (progress tracking),
and DynamicLifeline. None are deployed. The Basic engine is the public API; using it
is correct, not a gimping. Pro features are not required for fair throughput measurement.

### 8. Long-running worker does synthetic work, latency is measured
`oban_bench/lib/oban_bench/long_horizon.ex:471–502` (LongHorizonWorker) records
insertion-to-execution latency via `DateTime.diff(now, inserted_at)` (line 473) and
stores it in an ETS bag for percentile calculation. Work duration is controlled via
`JOB_WORK_MS` env var (default 1 ms, line 40). The latency measurement is inline,
not a separate polling loop; it is accurate and part of the documented contract
(long_horizon.ex:3–6, reference to benchmarks/portable/CONTRIBUTING_ADAPTERS.md).

## Recommendations
1. Override `PRODUCER_BATCH_MAX` to a value > 1 (e.g., 1000) when measuring producer
   throughput to exercise the documented bulk-insert path and prevent artificial
   row-at-a-time gimping.
2. Consider exercising `:error` or `:discard` returns in a dedicated scenario to
   measure retry and DLQ overhead; currently only the happy path (completion) is
   measured.
3. Verify `MAX_CONNECTIONS` matches the expected worker concurrency to avoid pool
   starvation; default 20 is conservative for high worker counts.
